## 🧩 0-2. 作るもの（家計簿アプリ）

元になっているのは React 1枚ファイルの「スマート家計簿」です。やれることは次の4つ。

| 機能            | 内容                                                   |
| --------------- | ------------------------------------------------------ |
| 認証            | 新規登録・ログイン・ログアウト（**パスワードはハッシュ化**） |
| ダッシュボード  | 今月の残高／収入／支出、支出の内訳、直近6ヶ月の推移      |
| 履歴            | 月ごとの記録一覧（削除できる）                         |
| 記録            | 支出・収入を新しく登録する                             |
| カテゴリー管理  | 支出/収入カテゴリーの追加・編集・削除（色も選べる）     |

React 版では Firebase（クラウドDB）に直接読み書きしていましたが、
Spring Boot 版では **自分のサーバー＋PostgreSQL** に置き換え、**層を分けて** 作ります。

## 🏛 0-3. レイヤード・アーキテクチャ（層構造）

このアプリは「役割ごとに住所（パッケージ）を分ける」のが基本です。

```text
[ ブラウザ ]
    │ HTTP（GET/POST）
    ▼
[ Controller ]  ← URLを受け取り、Serviceを呼び、画面を返す（薄い）
    │
    ▼
[  Service  ]   ← 業務の流れ・判断（ハッシュ化・集計・重複チェック）
    │
    ▼
[ Repository ]  ← DB操作だけ（SQLは自動生成）
    │
    ▼
[  Entity   ]   ← テーブル1行 ＝ クラス1個
    │
    ▼
[ PostgreSQL ]
```

加えて、画面とサーバーの境目に **Form**（画面の入力1件を表す箱）が入ります。

```text
[ 画面 ] ─入力─▶ [ Form ] ─Serviceが詰め替え─▶ [ Entity ] ─▶ [ DB ]
```

👉 **「画面の都合（Form）」と「DBの都合（Entity）」を混ぜない**。これが全体を貫く大原則です。

## 🔁 0-4. ★最重要★ なぜこの順番で作るのか

> **鉄則：「依存される側 → 依存する側」の順で作る**
> （A が B を import するなら、B を先に作る）

このアプリの依存関係を整理すると：

```text
TransactionType（enum）   ← 誰からも依存される（独立）
   ▲           ▲
User          Category    ← User は独立 / Category は User に依存
   ▲           ▲   ▲
   └─ Transaction ─┘      ← Transaction は User と Category の両方に依存
```

だから作る順番は：

| 順 | 作るもの        | 章   | 理由                                       |
| -- | --------------- | ---- | ------------------------------------------ |
| 1  | Entity          | 1〜3 | 全ての土台。依存される側から（型→User→Category→Transaction） |
| 2  | Repository      | 4    | Entity に依存する。Entityが揃ってからまとめて |
| 3  | Form            | 5    | 画面の入力箱。Service が使う                |
| 4  | Service         | 6    | Repository と Form の両方を使う。ハッシュ化もここ |
| 5  | Controller      | 7-1  | 一番上。全部に依存する                     |
| 6  | テンプレート＋Security | 7-2 | 画面と認可。最後に動く形にする            |

👉 この順で作ると **import エラーが出ず、IDE補完も効く**。行ったり来たりしません。

## 🔄 0-5. React版 → Spring Boot版 の対応表

| React版（1ファイル）                         | Spring Boot版（層に分割）                              |
| -------------------------------------------- | ------------------------------------------------------ |
| `useState` の `transactions`                 | `Transaction` Entity ＋ `transactions` テーブル        |
| `DEFAULT_CATEGORIES` ＋ カスタム             | `Category` Entity（利用者ごと・初期データを登録時に作成）|
| `appUser` / `registeredAccount`              | `User` Entity（**パスワードをBCryptでハッシュ化**）     |
| Firestore の読み書き（`onSnapshot`等）       | `Repository`（JpaRepository が SQL を自動生成）         |
| `handleAddTransaction` などの関数            | `Service` のメソッド（業務ロジック）                   |
| 画面の `formData` / `authForm`               | `Form` クラス（`@NotBlank` 等のバリデーション付き）     |
| `activeTab` での画面切り替え                 | `Controller` の URL マッピング＋Thymeleafテンプレート   |
| Firebase Auth                                | Spring Security（フォーム認証・CSRF・ログアウト）       |
| `useMemo` の集計（summary/categoryData等）   | `TransactionService` の集計メソッド＋DTO               |

👉 React では1ファイルに混ざっていた「画面・状態・通信・集計」が、
　 Spring Boot では **役割ごとに別ファイル** に分かれます。最初は増えて見えますが、
　 **どこに何を書くか迷わなくなる** のが層構造の利点です。

## 🔐 0-7. このアプリの「セキュリティの肝」

研修として特に意識してほしいポイントです。

1. **パスワードは必ずハッシュ化** する（`BCryptPasswordEncoder`）。生のまま保存しない。→ 第6章
2. **他人のデータを触らせない**。記録・カテゴリーは「ログイン中ユーザーのものか」を Service で必ずチェックする。
3. **ログイン中ユーザーはURLからではなく認証情報（`@AuthenticationPrincipal`）から取る**。→ 第7章
4. **削除は必ず POST**＋CSRFトークン（Thymeleafが自動挿入）。→ 第7章
