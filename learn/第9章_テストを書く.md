## 🧩 9-2. なぜテストを書くのか

> 自動テスト ＝ 「仕様どおり動くか」を **機械に毎回チェックさせる** 仕組み

手で画面を触って確認する「手動テスト」は、

- 変更のたびに全部やり直し（面倒・抜ける）
- 「ログイン後にダッシュボードが落ちる」ような不具合に **気づくのが遅れる**

自動テストがあれば、`mvnw test` の1コマンドで**何度でも同じ確認**ができる。
リファクタや機能追加のあとに走らせれば、**壊れた箇所をその場で**教えてくれる。

```text
[ コードを変更 ] → [ mvnw test ] → 緑（成功）なら安心 / 赤（失敗）なら原因がすぐ分かる
```

## 🗂️ 9-3. テストの種類（テストピラミッド）

```text
        ▲ 少なく・遅い
        │   E2E（画面を実ブラウザで操作）   ← 今回は対象外
        │   結合テスト（@SpringBootTest で全部起動）
        │   単体テスト（層を1つずつ・モックで隔離）   ← ★この章の主役★
        ▼ 多く・速い
```

- **単体テスト（Unit Test）**：1つのクラス／層だけを取り出し、**まわりはニセモノ（モック）**にして検証する。速い・原因が特定しやすい。
- **結合テスト（Integration Test）**：`@SpringBootTest` で全部を本物どおり起動して検証する。リアルだが遅い。

この章では **層ごとの単体テスト** を中心に書く。
（既存の `ExpenseTrackerAppApplicationTests`＝`@SpringBootTest` が結合テストの最小例。）

## 🎭 9-4. モック（Mock）とは

> モック ＝ 本物のフリをする **ニセモノの部品**。指示した通りの値を返す／呼ばれた回数を覚える

例：`TransactionController` は `TransactionService` を使う。
Controller だけをテストしたいので、Service は **モック**にして
「`summarize()` が呼ばれたら、こういう集計結果を返したことにする」と**台本**を渡す。

```java
// userService.findByEmail("ake@test.com") が呼ばれたら、用意した user を返す（台本）
when(userService.findByEmail("ake@test.com")).thenReturn(user);
```

これで DB もパスワード照合も走らせずに、**Controller の振る舞いだけ**を切り出して試せる。

### Mockito の基本5パターン

| やりたいこと | 書き方 |
| --- | --- |
| 戻り値を仕込む | `when(mock.method(引数)).thenReturn(値)` |
| 例外を投げさせる | `when(mock.method(...)).thenThrow(new XxxException())` |
| 戻り値が void で例外 | `doThrow(new XxxException()).when(mock).method(...)` |
| 呼ばれたか検証 | `verify(mock).method(...)` ／ 呼ばれていない `verify(mock, never()).method(...)` |
| 渡された引数を捕まえる | `ArgumentCaptor<T>` ＋ `verify(mock).save(captor.capture())` |

「引数は何でもよい」を表すのが `any()` / `eq(値)`（`org.mockito.ArgumentMatchers`）。

## 🧪 9-5. 3層それぞれのテスト戦略

| 層 | 起動するもの | 本物 | モック | 使うアノテーション |
| --- | --- | --- | --- | --- |
| Controller | MVC＋Security だけ | Controller | Service | `@WebMvcTest` ＋ `@MockitoBean` |
| Service | 何も起動しない | Service | Repository / 他Service | `@ExtendWith(MockitoExtension.class)` ＋ `@Mock` / `@InjectMocks` |
| Dao(Repository) | JPA＋H2 だけ | Repository＋本物のDB(H2) | （無し） | `@DataJpaTest` ＋ `TestEntityManager` |

ポイントは **「検証したい層だけ本物、まわりはニセモノ」**。
Service は集計ロジックが主役なので DB を本物にする必要がない（モックで十分）。
逆に Repository は「SQLが正しいか」が主役なので **DB は本物（H2）**にする。

## 🎬 9-6. テストの型 ― Given / When / Then（AAA）

どのテストも、頭の中はこの3段で書くと読みやすい。

```text
Given（準備）: テストデータ・モックの台本を用意する
When （実行）: テスト対象のメソッド／リクエストを1回だけ実行する
Then （検証）: 結果（戻り値・ステータス・呼び出し）が期待どおりか確かめる
```

例（Service の登録テスト）：

```java
// Given: 入力フォームと「種類一致のカテゴリー」を用意
TransactionForm form = ...;
when(categoryService.findOwnedById(user, 10L)).thenReturn(food);
when(transactionRepository.save(any())).thenAnswer(inv -> inv.getArgument(0));
// When: 登録を実行
Transaction saved = transactionService.register(user, form);
// Then: save に渡された中身が Form どおりか検証
ArgumentCaptor<Transaction> captor = ArgumentCaptor.forClass(Transaction.class);
verify(transactionRepository).save(captor.capture());
assertEquals(1000, captor.getValue().getAmount());
```

## 🛡️ 9-7. Controller テスト（`@WebMvcTest`）の要点

対象ソース：
[../src/test/java/com/example/expensetracker/controller/](../src/test/java/com/example/expensetracker/controller/)（6クラス）

### ① 何が起動するか
`@WebMvcTest(対象Controller.class)` は **Web（MVC）層だけ**を起動する。
Service や Repository の本物は読み込まれない。だから依存 Service は `@MockitoBean` で用意する。

```java
@WebMvcTest(DashboardController.class)   // この Controller だけ
@Import(SecurityConfig.class)            // 本物の認可ルールも使う
class DashboardControllerTest {
    @Autowired MockMvc mockMvc;          // 擬似HTTPを送る道具
    @MockitoBean TransactionService transactionService; // 依存はモック
    @MockitoBean UserService userService;
}
```

### ② MockMvc で擬似リクエストを送る
```java
mockMvc.perform(get("/dashboard"))          // GET /dashboard を実行
       .andExpect(status().isOk())          // 200 か？
       .andExpect(view().name("dashboard")) // ビュー名は dashboard か？
       .andExpect(model().attributeExists("summary", "breakdown", "trend")); // model は揃ったか？
```

### ③ Spring Security と一緒に使うときの2点セット
- **ログイン済みにする** … `@WithMockUser(username = "ake@test.com")`
  - `@AuthenticationPrincipal UserDetails principal` の `getUsername()` がこの値になる。
- **POST には CSRF トークン** … `.with(csrf())`
  - このアプリは CSRF 保護が有効なので、付けないと 403 で弾かれる。

```java
@WithMockUser(username = "ake@test.com")
void register_valid() throws Exception {
    mockMvc.perform(post("/transactions")
                    .param("type", "EXPENSE").param("amount", "1000")
                    .param("transactionDate", "2026-06-15").param("categoryId", "10")
                    .with(csrf()))                              // ★CSRF★
           .andExpect(redirectedUrl("/transactions?month=2026-06"))
           .andExpect(flash().attributeExists("message"));
    verify(transactionService).register(eq(user), any());      // 委譲されたか
}
```

### ④ 認可（未ログイン）の確認もできる
`@Import(SecurityConfig.class)` で本物のルールを使うので、
「未ログインで `/dashboard` → `/login` へ」も検証できる（`@WithMockUser` を付けないだけ）。

```java
mockMvc.perform(get("/dashboard"))
       .andExpect(status().is3xxRedirection())
       .andExpect(redirectedUrl("/login"));   // ★Security 7 は相対パス /login★
```

## 🧮 9-8. Service テスト（純粋 Mockito）の要点

対象ソース：
[../src/test/java/com/example/expensetracker/service/TransactionServiceTest.java](../src/test/java/com/example/expensetracker/service/TransactionServiceTest.java)

Spring を起動しないので **一番速い**。`@Mock` の依存を `@InjectMocks` の本物へ差し込む。

```java
@ExtendWith(MockitoExtension.class)
class TransactionServiceTest {
    @Mock TransactionRepository transactionRepository; // DBはモック
    @Mock CategoryService categoryService;
    @InjectMocks TransactionService transactionService; // 本物（上2つを注入済み）
}
```

検証の主役は **集計ロジック**：

- `summarize` … 収入/支出を仕分けして残高を出すか
- `expenseBreakdown` … 支出だけをカテゴリー別に合計し、割合付き・降順か
- `recentTrend` … 6ヶ月分・古い順・月ラベルが正しいか
- `register` … Form を Entity に詰め替え、**持ち主は必ず user**、種類不一致は例外か
- `delete` … 他人の記録・存在しないIDは例外で弾くか

Repository の戻り値は `when(...).thenReturn(リスト)` で自由に作れるので、
「収入1000・支出500のとき残高は500」のような**境界づくり**が一瞬でできる。

## 🗄️ 9-9. Dao(Repository) テスト（`@DataJpaTest` ＋ H2）の要点

対象ソース：
[../src/test/java/com/example/expensetracker/repository/TransactionRepositoryTest.java](../src/test/java/com/example/expensetracker/repository/TransactionRepositoryTest.java)

Repository は「メソッド名から作られる SQL が正しいか」が主役。
ここは **本物のDB** がないと意味がないが、本番の PostgreSQL を使うと準備が大変。
→ **テスト用の H2（メモリDB）** を使う。

```java
@DataJpaTest   // JPA層だけ起動（Controller/Serviceは読まない）
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE) // H2設定をそのまま使う
class TransactionRepositoryTest {
    @Autowired TransactionRepository transactionRepository;
    @Autowired TestEntityManager entityManager; // テストデータを直接INSERTする道具
}
```

### H2 を有効にする2つの準備
1. **pom.xml** に H2 を `test` スコープで追加（[../pom.xml](../pom.xml)）
   ```xml
   <dependency><groupId>com.h2database</groupId><artifactId>h2</artifactId><scope>test</scope></dependency>
   ```
2. **テスト用設定** [../src/test/resources/application.properties](../src/test/resources/application.properties)
   - テスト時はこのファイルが本番設定を **丸ごと上書き** する。
   - `jdbc:h2:mem:testdb` ／ `ddl-auto=create-drop`（Entityからテーブル自動生成→破棄）。

### テストデータは TestEntityManager で投入
```java
entityManager.persist(user);              // 先に持ち主を入れて採番
entityManager.persist(food);              // カテゴリー
entityManager.persist(newTx(LocalDate.of(2026,6,20), 700, transport));
entityManager.flush();                    // INSERTをDBへ反映
entityManager.clear();                    // 1次キャッシュを空にし、次のSELECTを本物にする
```

`@DataJpaTest` は各テスト後に **自動ロールバック** するので、データは残らない（テスト同士が汚し合わない）。

## ⚠️ 9-10. Spring Boot 4.0 でハマるポイント（重要）

このアプリは **Spring Boot 4.0 / Spring Framework 7 / Spring Security 7** で動く。
3.x までの記事どおりに書くと**コンパイルや実行で詰まる**箇所がある。

| 項目 | 3.x まで | ★4.0（このアプリ）★ |
| --- | --- | --- |
| `@WebMvcTest` のパッケージ | `org.springframework.boot.test.autoconfigure.web.servlet` | **`org.springframework.boot.webmvc.test.autoconfigure`** |
| `@DataJpaTest` のパッケージ | `...boot.test.autoconfigure.orm.jpa` | **`...boot.data.jpa.test.autoconfigure`** |
| `@AutoConfigureTestDatabase` | `...boot.test.autoconfigure.jdbc` | **`...boot.jdbc.test.autoconfigure`** |
| `TestEntityManager` | `...boot.test.autoconfigure.orm.jpa` | **`...boot.jpa.test.autoconfigure`** |
| Bean をモック差し替え | `@MockBean`（`boot.test.mock.mockito`） | **`@MockitoBean`**（`org.springframework.test.context.bean.override.mockito`）★`@MockBean`は廃止★ |
| 未認証リダイレクト先 | `http://localhost/login`（絶対） | **`/login`（相対）** |

> MockMvc 本体（`org.springframework.test.web.servlet.*`）・`@WithMockUser`・`csrf()` の場所は 4.0 でも変わらない。

## ▶️ 9-11. テストの実行方法

```bash
# 全テストを実行（src/test 以下すべて）
./mvnw test

# 1クラスだけ実行
./mvnw test -Dtest=TransactionServiceTest

# 名前パターンで実行（全Controllerテスト）
./mvnw test -Dtest='*ControllerTest'
```

> 補足：このプロジェクトは Java 17 以上で動く。`JAVA_HOME` が古いJDKを指していると Wrapper が動かないので、インストール済みのJDK（例：jdk-22）に合わせる。

成功すると最後にこう出る：

```text
Tests run: 34, Failures: 0, Errors: 0, Skipped: 0
BUILD SUCCESS
```
