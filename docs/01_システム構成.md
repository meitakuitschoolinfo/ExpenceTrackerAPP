## 1-2. レイヤード・アーキテクチャ

```mermaid
flowchart TB
    browser([ブラウザ])

    subgraph controller[Controller 層 ＝ HTTPの受付]
        direction LR
        ctl[各Controller]
    end
    subgraph service[Service 層 ＝ 業務ロジック]
        direction LR
        svc[各Service]
    end
    subgraph repo[Repository 層 ＝ DB操作]
        direction LR
        rep[各Repository]
    end
    subgraph entity[Entity ＝ テーブル1行]
        direction LR
        ent[User / Category / Transaction]
    end

    browser -->|GET/POST| controller
    controller -->|呼び出し| service
    service -->|呼び出し| repo
    repo -->|JPA| entity
    entity -->|JDBC| db[(PostgreSQL)]

    form[Form ＝ 画面入力の箱]:::side -.->|"@Valid"| controller
    dto[DTO ＝ 集計結果の箱]:::side -.->|model| controller
    classDef side fill:#eef,stroke:#88a;
```

| 層 | 責務 | やらないこと |
| -- | ---- | ------------ |
| Controller | URL受付・`@Valid`発動・Service呼び出し・画面名/リダイレクト返却 | SQL・業務ルール・HTML組み立て |
| Service | 業務判断・詰め替え・ハッシュ化・集計・トランザクション境界 | HTTP・SQL |
| Repository | DB入出力（CRUD・検索） | if/for/業務判定 |
| Entity | テーブル1行の「形」 | 業務ロジック |
| Form | 画面入力1件の保持＋入力検証 | DB制約 |
| DTO | 集計結果を画面へ運ぶ | DBと結合 |

## 1-4. 実行構成・環境設定

| 設定キー | 値 | 意味 |
| -------- | -- | ---- |
| `spring.datasource.url` | `jdbc:postgresql://localhost:5433/expense_tracker` | 接続先DB |
| `spring.datasource.username` | `postgres` | DBユーザー |
| `spring.datasource.password` | `123456` | DBパスワード（環境に合わせ変更） |
| `spring.jpa.hibernate.ddl-auto` | `update` | 起動時にEntityからテーブルを差分更新（学習用） |
| `spring.jpa.show-sql` | `true` | 実行SQLをログ出力 |
| `server.port` | `8080` | 待ち受けポート |

### 起動手順

```bash
# 1. DB作成（初回のみ）
psql -U postgres -c "CREATE DATABASE expense_tracker;"

# 2. 起動（Windows）
mvnw.cmd spring-boot:run

# 3. アクセス
#   http://localhost:8080/  → 未ログインなら /login へ
```

> 📌 `ddl-auto=update` は学習用。本番は `validate` ＋ Flyway/Liquibase を推奨。

## 1-5. 処理の一往復（リクエストの流れ）

```mermaid
sequenceDiagram
    participant B as ブラウザ
    participant SF as Security Filter
    participant C as Controller
    participant S as Service
    participant R as Repository
    participant DB as PostgreSQL

    B->>SF: GET /dashboard
    SF->>SF: 認証チェック（未ログインなら /login へ）
    SF->>C: 認証OK → 委譲（principal付き）
    C->>S: 集計を依頼（user, month）
    S->>R: 月範囲の記録を取得
    R->>DB: SELECT ... BETWEEN
    DB-->>R: 記録リスト
    R-->>S: List<Transaction>
    S-->>C: DTO（summary/breakdown/trend）
    C-->>B: dashboard.html（Thymeleafで描画）
```

## 1-6. シーケンス図（記録登録の一連の流れ）

代表的な更新系ユースケースとして「記録の登録（FNC-09）」を、ブラウザ〜DBの往復で示す。
※詳細な条件分岐は [06_処理設計 6-4](./06_処理設計.md) を参照。

```mermaid
sequenceDiagram
    autonumber
    participant B as ブラウザ
    participant SF as Security Filter
    participant C as TransactionController
    participant TS as TransactionService
    participant CS as CategoryService
    participant TR as TransactionRepository
    participant DB as PostgreSQL

    B->>SF: POST /transactions（フォーム＋CSRF）
    SF->>SF: 認証・CSRF を検証
    SF->>C: @Valid TransactionForm を委譲
    alt 入力エラー（BindingResult.hasErrors）
        C-->>B: form.html を再描画（項目エラー表示）
    else 入力OK
        C->>TS: register(user, form)
        TS->>CS: findOwnedById(user, categoryId)
        CS->>DB: SELECT category（user一致で持ち主確認）
        DB-->>CS: Category
        CS-->>TS: Category（持ち主OK）
        TS->>TS: category.type == form.type を検証
        TS->>TR: save(Transaction)（user＝ログイン固定）
        TR->>DB: INSERT INTO transactions
        DB-->>TR: OK
        TR-->>TS: 保存済 Transaction
        TS-->>C: 完了
        C-->>B: redirect /transactions?month=（PRG＋フラッシュ）
    end
```

## 1-7. アクティビティ図（利用者の操作フロー）

アプリ利用開始からログアウトまでの、利用者から見た行動の流れ。ひし形は判断、角丸は開始/終了。

```mermaid
flowchart TD
    start([開始]) --> access["トップ「/」へアクセス"]
    access --> isAuth{ログイン済み?}

    isAuth -- いいえ --> login["ログイン画面"]
    login --> hasAcc{アカウント有り?}
    hasAcc -- なし --> reg["新規登録<br/>（初期カテゴリー12件を自動作成）"]
    reg --> login
    hasAcc -- あり --> cred["email・パスワードを入力"]
    cred --> authOK{認証成功?}
    authOK -- 失敗 --> login
    authOK -- 成功 --> dash

    isAuth -- はい --> dash["ダッシュボード<br/>残高・支出内訳・推移を表示"]

    dash --> op{操作を選択}
    op -->|＋記録| input["種類・日付・金額・<br/>カテゴリー・メモを入力"]
    input --> vOK{入力OK?}
    vOK -- NG --> input
    vOK -- OK --> owner{自分のカテゴリー<br/>かつ種類一致?}
    owner -- いいえ --> input
    owner -- はい --> save["記録を保存（PRG）"]
    save --> list["履歴一覧"]

    op -->|履歴| list
    op -->|設定| cat["カテゴリー 追加・編集・削除"]
    op -->|ログアウト| done

    list --> op
    cat --> op
    done([終了])
```
